# 12 任务图 - Task System 实现汇总

本文档记录了 my_agent 项目中任务图功能的完整实现过程。

---

## 功能概述

### 核心特性

1. **持久化任务存储**：每个任务保存为独立的 JSON 文件（`.tasks/<task_id>.json`）
2. **DAG 依赖关系**：通过 `blockedBy` 字段声明任务依赖，支持复杂的任务依赖图
3. **任务生命周期**：`pending` → `in_progress` → `completed`
4. **自动依赖检查**：`can_start()` 检查所有依赖是否完成
5. **下游任务解锁**：完成任务时自动报告被解锁的下游任务

### 工作流程

```
1. create_task(subject, description, blockedBy) 
   → 生成 ID (时间戳 + 自增序列)
   → 保存到 .tasks/<task_id>.json

2. list_tasks() 
   → 扫描 .tasks/ 目录
   → 按序列号排序返回

3. claim_task(task_id)
   → 检查 status = "pending"
   → 检查 can_start() = True
   → 设置 owner，status = "in_progress"

4. complete_task(task_id)
   → 设置 status = "completed"
   → 扫描所有任务，找出被解锁的下游任务
   → 返回解锁列表
```

### 解决的问题

- **任务顺序管理**：避免"先盖屋顶再打地基"的问题
- **跨会话持久化**：任务保存在磁盘，重启后可继续
- **多 agent 协作**：通过 `owner` 字段记录任务所有者
- **进度追踪**：清晰的任务状态和依赖关系

#### ID 生成（时间戳 + 自增序列）

```python
HIGHWATERMARK_FILE = TASKS_DIR / ".highwatermark"

def _next_task_id() -> str:
    """生成下一个任务 ID（时间戳 + 自增序列）
    
    特点：
    - 时间戳：便于调试，能看出创建时间
    - 序列号：保证唯一性，即使删除任务也不重用
    - 格式：task_<timestamp>_<sequence>（4位补零）
    
    示例：task_1720512345_0001
    """
    try:
        sequence = int(HIGHWATERMARK_FILE.read_text(encoding="utf-8").strip())
    except (FileNotFoundError, ValueError):
        sequence = 0
    
    next_sequence = sequence + 1
    HIGHWATERMARK_FILE.write_text(str(next_sequence), encoding="utf-8")
    
    timestamp = int(time.time())
    return f"task_{timestamp}_{next_sequence:04d}"
```

**设计亮点**：
- ✅ 时间戳：能看出创建时间，便于调试
- ✅ 序列号：通过 `.highwatermark` 文件保证唯一性和防重用
- ✅ 可读性：`task_1720512345_0001` 比纯数字更清晰
- ✅ 简单：单进程场景无需文件锁

#### 依赖检查

```python
def can_start(task_id: str) -> bool:
    """检查任务是否可以开始（所有依赖是否都完成）
    
    边缘情况处理：
    - 缺失的依赖（文件不存在）视为 blocked
    - 避免因为引用错误 ID 而崩溃
    """
    task = load_task(task_id)
    for dep_id in task.blockedBy:
        if not _task_path(dep_id).exists():
            return False  # 依赖不存在，视为 blocked
        dep = load_task(dep_id)
        if dep.status != "completed":
            return False  # 依赖未完成
    return True
```

#### 完成任务并解锁下游

```python
def complete_task(task_id: str) -> str:
    """完成任务并解锁下游任务
    
    下游任务解锁逻辑：
    - 当前任务完成后，依赖它的任务可能变为可开始状态
    - 扫描所有 pending 任务，找出现在 can_start() 为 True 的
    - 提示用户哪些任务现在可以认领了
    """
    task = load_task(task_id)
    
    if task.status != "in_progress":
        return f"Task {task_id} is {task.status}, cannot complete"
    
    task.status = "completed"
    save_task(task)
    
    # 找出被解锁的下游任务
    unblocked = [t.subject for t in list_tasks()
                 if t.status == "pending" and t.blockedBy and can_start(t.id)]
    
    print(f"  \033[32m[complete] {task.subject} [OK]\033[0m")
    
    msg = f"Completed {task_id} ({task.subject})"
    if unblocked:
        msg += f"\nUnblocked: {', '.join(unblocked)}"
        print(f"  \033[33m[unblocked] {', '.join(unblocked)}\033[0m")
    
    return msg
```

## 实现细节与巧思

### 1. ID 生成方案选择

对比了三种方案：

| 方案 | 优点 | 缺点 | 采用 |
|------|------|------|------|
| 时间戳 + 随机数 | 简单，无需维护状态 | 理论上可能冲突（极低概率） | ❌ |
| 顺序 ID + highwatermark | 严格唯一，简洁 | 需要维护文件，单进程场景下也需要 | ❌ |
| **时间戳 + 自增序列** | 兼具可读性和唯一性 | 需要维护 highwatermark | ✅ |

**最终选择**：时间戳 + 自增序列
- 时间戳部分：便于调试和查看创建时间
- 序列号部分：保证唯一性和防重用
- 示例：`task_1720512345_0001`

### 2. 依赖检查的防御性设计


## 与其他系统的关系

### Task System vs TodoWrite

| | TodoWrite (s05) | Task System (s12) |
|---|---|---|
| **定位** | 当前任务的执行清单 | 可恢复的任务系统 |
| **存储** | 进程内 / 会话状态 | `.tasks/{id}.json` |
| **依赖** | 无 | `blockedBy` / `blocks` 依赖图 |
| **生命周期** | 当前会话 / 当前任务 | 跨会话保留 |
| **分工** | 不负责任务认领 | `owner` / claim |
| **状态** | pending / in_progress / completed | pending / in_progress / completed |
| **粒度** | Agent 自己的步骤 | 可被认领、追踪、解锁的任务 |

**结论**：Task System 和 TodoWrite 是互补的，不是替代关系。

### 状态机

```
pending ──claim──→ in_progress ──complete──→ completed
```

- **claim_task**: `pending` → `in_progress`，设置 owner，开始工作
- **complete_task**: `in_progress` → `completed`，标记完成，解锁下游

**注意**：CC 实际实现中有 `release` 路径（`in_progress` → `pending`），用于 teammate 终止或 shutdown 时重置任务。教学版省略了这一恢复路径。

## 未来改进方向

基于真实 CC 实现的改进建议：

### 1. 并发安全

**当前实现**：单进程，无文件锁

**CC 实现**：
```typescript
// 使用 proper-lockfile 锁住任务文件
await lockfile.lock(taskPath, { retries: 30, maxTimeout: 100 });
try {
  // 重新读取任务（防 TOCTOU）
  const task = await loadTask(taskId);
  // ... 修改任务 ...
  await saveTask(task);
} finally {
  await lockfile.unlock(taskPath);
}
```

**改进点**：
- 使用 `filelock` 或 `fcntl` 实现文件锁
- 防止多进程同时修改同一个任务

### 2. 完整的 TaskRecord 字段

**当前实现**：6 个字段（id, subject, description, status, owner, blockedBy）

**CC 实现**：9 个字段
```typescript
interface TaskRecord {
  id: string;
  subject: string;
  description: string;
  activeForm?: string;  // 进行时态，在 spinner 显示
  owner?: string;
  status: "pending" | "in_progress" | "completed";
  blocks: string[];     // 此任务阻塞的任务 ID（下游）
  blockedBy: string[];  // 阻塞此任务的任务 ID（上游）
  metadata?: Record<string, any>;  // 任意扩展键值对
}
```

**改进点**：
- 添加 `activeForm`：显示在 spinner 中的进行时态
- 添加 `blocks`：双向依赖关系，便于查询
- 添加 `metadata`：扩展字段，存储任意键值对

### 3. 环检测

**当前实现**：无环检测（教学版假设用户不会创建环）

**CC 实现**：
```typescript
function detectCycle(taskId: string, visited: Set<string>): boolean {
  if (visited.has(taskId)) return true;
  visited.add(taskId);
  
  const task = loadTask(taskId);
  for (const depId of task.blockedBy) {
    if (detectCycle(depId, visited)) return true;
  }
  
  visited.delete(taskId);
  return false;
}
```

**改进点**：
- 在 `create_task()` 时检测是否会形成环
- 拒绝创建会导致环的依赖关系

### 4. Release 路径

**当前实现**：无法将 `in_progress` 任务重置为 `pending`

**CC 实现**：
```typescript
function releaseTask(taskId: string) {
  const task = loadTask(taskId);
  if (task.status !== "in_progress") return;
  
  task.status = "pending";
  task.owner = null;  // 清除 owner
  saveTask(task);
}
```

**改进点**：
- 添加 `release_task()` 函数
- 用于 agent 终止或 shutdown 时重置未完成的任务

### 5. 工具自动注册

**当前实现**：手动在 `Tools/tools.py` 中添加工具定义和映射

**改进方案**：
```python
# tasks.py 末尾

    "name": "create_task",
    "description": "...",
    "input_schema": {...}
```

**改进点**：
- 工具模块自己注册工具，无需手动修改 `Tools/tools.py`
- 新增工具时只需创建新模块并调用 `register_tool()`
