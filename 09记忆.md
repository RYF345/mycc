## 功能概述

Memory 功能实现了**跨会话、跨压缩的知识存储系统**，解决了 autoCompact 会丢失用户偏好细节的问题。

### 核心特性

- ✅ **持久化存储**：`.memory/` 目录，Markdown + YAML frontmatter
- ✅ **智能加载**：LLM side-query 选择相关记忆，降级到关键词匹配
- ✅ **自动提取**：每轮对话结束时从压缩前快照提取新记忆
- ✅ **定期整理**：文件数达到阈值（15）时自动合并去重
- ✅ **四种类型**：user（用户偏好）、feedback（反馈约束）、project（项目背景）、reference（外部资源）

### 工作流程

```
用户输入
  ↓
1. load_memories() - 选择相关记忆（LLM side-query）
  ↓
2. build_system() - 将记忆索引注入 SYSTEM prompt
  ↓
3. 压缩管线 - snip/micro/tool_result_budget/compact_history
  ↓
4. 临时注入记忆到当前 user turn
  ↓
5. LLM 调用
  ↓
6. 对话结束 → extract_memories() → consolidate_memories()
```

```python
try:
```

```python
    request_messages = messages
    if memories_content and memory_turn is not None and memory_turn < len(messages):
```

**业务逻辑**：

- **临时注入**：记忆内容只在 LLM 请求中出现，不污染 messages 历史
- **浅拷贝**：`messages.copy()` 只拷贝列表，不拷贝内部字典，节省内存

- `consolidate_memories()`：检查文件数是否达到阈值，触发整理

**关键巧思 - 压缩前快照**：
```python
# 为什么需要快照？
# 1. 每轮开始时保存
pre_compress = [m.copy() if isinstance(m, dict) else m for m in messages]

# 2. 然后压缩 messages
messages[:] = snip_compact(messages)  # 裁掉中间消息
messages[:] = micro_compact(messages)  # 工具结果占位

# 3. LLM 调用使用压缩后的 messages（节省 token）

# 4. 提取记忆使用 pre_compress（保留完整细节）
extract_memories(pre_compress)  # ← 能看到被压缩掉的用户偏好
```

**为什么不恢复快照？**
- 快照只是"拍照留底"，用于读取
- `messages` 永远保持压缩状态（节省成本）
- `pre_compress` 读取一次后就丢弃

## 实现细节与巧思

### 1. 两条路径加载记忆

**路径一：索引常驻 SYSTEM prompt（便宜）**
```python
# build_system() 中
memory_index = MEMORY_INDEX.read_text(encoding="utf-8")
return f"...Memory Index:\n{memory_index}\n"
```
- 只包含 name + description（轻量）
- 每次都加载，让 LLM 知道有哪些记忆
- 可以被 prompt cache 缓存

**路径二：相关记忆按需注入（按需）**
```python
# agent_loop() 开始时
memories_content = load_memories(messages)  # LLM side-query 选择
# LLM 调用时临时注入
request_messages[memory_turn]["content"] = memories_content + "\n\n" + original_content
```
- 只加载相关的（最多 5 条）
- 包含完整内容（body）
- 临时注入，不永久保存

### 2. LLM side-query + 关键词匹配降级

**主策略：LLM side-query**
```python
# 构建记忆目录
catalog = "0: user-preference-tabs — User prefers tabs\n1: feedback-no-mock — ..."

# 让 LLM 选择
prompt = f"""Select relevant memory indices...
Recent conversation: {recent_text}
Memory catalog: {catalog}
"""

response = client.messages.create(...)  # 返回 [0, 2]
```

**降级策略：关键词匹配**
```python
try:
    # LLM side-query
    selected = select_relevant_memories(messages)
except Exception:
    # 降级到关键词匹配
    selected = select_relevant_memories_fallback(messages)
```

**为什么这样设计？**
- LLM 理解语义，更准确
- 关键词匹配作为保底，确保鲁棒性
- API 失败不影响主流程

### 3. 提取 ≠ 整理

**提取（extract_memories）**：生产者
- 每轮对话结束时运行
- 从对话中识别新偏好
- 创建新文件

**整理（consolidate_memories）**：清洁工
- 文件数达到阈值（15）时触发
- 合并重复、删除过时
- 减少文件数

```python
if response.stop_reason != "tool_use":
    extract_memories(pre_compress)  # 可能增加文件
    consolidate_memories()           # 达到阈值才触发整理
```

```python
def should_extract_memories(messages):
    """判断是否应该提取记忆"""
    # 1. 对话太短，不提取
    if len(messages) < 3:
        return False
```

### 6. Dream 门控优化

**当前**：只有文件数阈值

**CC 做法**：四层门控（`autoDream.ts:130-190`）
1. **时间门控**：距上次合并 ≥ 24 小时
2. **扫描节流**：避免频繁扫描文件系统
3. **会话门控**：修改了 ≥ 5 个会话 transcript
4. **锁门控**：没有其他进程正在合并

**改进**：
```python
CONSOLIDATE_THRESHOLD = 15
CONSOLIDATE_MIN_INTERVAL = 24 * 3600  # 24 小时
_last_consolidate_time = 0

def consolidate_memories():
    """整理记忆：四层门控"""
    global _last_consolidate_time
    
    files = list_memory_files()
    
    # 门控 1: 文件数阈值
    if len(files) < CONSOLIDATE_THRESHOLD:
        return
    
    # 门控 2: 时间间隔
    now = time.time()
    if now - _last_consolidate_time < CONSOLIDATE_MIN_INTERVAL:
        return
    
    # 门控 3: 锁文件（避免并发）
    lock_file = MEMORY_DIR / ".consolidate-lock"
    if lock_file.exists():
        # 检查锁是否过期（1 小时）
        if now - lock_file.stat().st_mtime < 3600:
            return
    
    try:
        # 创建锁
        lock_file.touch()
        
        # 执行整理
        # ...
        
        _last_consolidate_time = now
    finally:
        # 删除锁
        lock_file.unlink(missing_ok=True)
```

```python
    summary = summarize_session(messages)
    
    (session_dir / "memory.md").write_text(summary, encoding="utf-8")

def load_session_memory():
    """加载会话记忆"""
    session_file = SESSION_MEMORY_DIR / SESSION_ID / "memory.md"
    if session_file.exists():
        return session_file.read_text(encoding="utf-8")
    return ""

# 在 compact_history() 中
def compact_history(msgs):
    # 保存会话记忆
    save_session_memory(msgs)
    
    # 读取会话记忆
    session_memory = load_session_memory()
    
    # 如果有会话记忆，用它做摘要基础
    if session_memory and len(session_memory) > 1000:
        summary = f"[Session context]\n{session_memory}\n\n[Recent updates]\n{recent_summary}"
    else:
        summary = summarize_history(msgs)
    
    return [{"role": "user", "content": f"[Compacted]\n\n{summary}"}]
```

## 附录：memory_turn 详解

### 问题：为什么需要 memory_turn？

很多人会疑惑：**最后一条不就是 user 消息吗，为什么还要特地标记索引？**

答案：90% 的情况确实如此，但 `memory_turn` 是**防御性设计**，处理边缘情况。

---

### memory_turn 的计算（第 75 行）

```python
memory_turn = len(messages) - 1 if messages and isinstance(messages[-1].get("content"), str) else None
```

**逻辑拆解**：
```python
if messages:  # 检查非空
    if isinstance(messages[-1].get("content"), str):  # 检查是字符串
        memory_turn = len(messages) - 1  # 最后一条的索引
    else:
        memory_turn = None  # 不是字符串，不注入
else:
    memory_turn = None  # 空列表，不注入
```

**返回值**：
- **正常情况**：最后一条消息的索引（例如：3 条消息 → `memory_turn = 2`）
- **边缘情况**：`None`（列表为空，或最后一条是工具结果）

---

### memory_turn 的使用（第 105-110 行）

```python
# LLM 调用时
request_messages = messages
if memories_content and memory_turn is not None and memory_turn < len(messages):
    request_messages = messages.copy()
    request_messages[memory_turn] = {
        **messages[memory_turn],
        "content": memories_content + "\n\n" + messages[memory_turn]["content"],
    }
```

**三层检查**：
1. `memories_content` 不为空（有相关记忆）
2. `memory_turn is not None`（有有效的注入位置）
3. `memory_turn < len(messages)`（索引有效，防止越界）

**只有全部满足，才会注入记忆**。

---

### 为什么不能直接用 messages[-1]？

#### 边缘情况 1：messages 为空（第一轮）

```python
# 第一次调用
agent_loop([])

# 如果直接用 messages[-1]
messages[-1]  # ❌ IndexError: list index out of range

# 用 memory_turn
memory_turn = None  # ✅ 安全，跳过注入
```

#### 边缘情况 2：最后一条是工具结果（列表格式）

```python
messages = [
    {"role": "user", "content": "Run ls"},
    {"role": "assistant", "content": [{"type": "tool_use", ...}]},
    {"role": "user", "content": [                    # ← 最后一条，但是列表！
        {"type": "tool_result", "content": "file1.txt"}
    ]},
]

# 检查类型
isinstance(messages[-1].get("content"), str)  # False
memory_turn = None  # ← 不注入

# 如果强行注入
messages[-1]["content"] = memories + "\n\n" + messages[-1]["content"]
#                                            ^^^^^^^^^^^^^^^^^^^^^^
#                                            列表，不能和字符串拼接！❌
```

#### 边缘情况 3：压缩后索引失效（理论上）

```python
# 假设压缩管线改变了 messages 长度
memory_turn = 5  # 原来有 6 条消息

# 压缩后只剩 3 条
messages = [...]  # len(messages) = 3

# 第三层检查
if memory_turn < len(messages):  # 5 < 3 → False
    # 不执行，避免越界 ✅
```

**注意**：在当前实现中，这种情况不会发生，因为 `memory_turn` 在循环外计算（第 75 行），压缩在循环内执行（第 86 行）。但这层检查提供了额外的安全保障。

### 为什么用索引而不是直接修改？

**方案 A：直接修改（❌ 会污染原始 messages）**
```python
messages[-1]["content"] = memories + "\n\n" + messages[-1]["content"]
# 问题：原始 messages 被修改了，下次循环会重复注入记忆
```

**方案 B：用索引定位，复制后修改（✅ 不污染原始）**
```python
memory_turn = len(messages) - 1
request_messages = messages.copy()  # 浅拷贝列表
request_messages[memory_turn] = {   # 只修改副本
    **messages[memory_turn],
    "content": memories + "\n\n" + messages[memory_turn]["content"],
}
# 原始 messages 保持不变
```

---

### 设计原则

`memory_turn` 体现了三个原则：

1. **防御性编程**：处理边缘情况（空列表、工具结果、索引变化）
2. **类型安全**：确保 content 是字符串再拼接
3. **清晰表达**：明确"注入到哪一条"，而不是隐式的 `messages[-1]`

**形象比喻**：
- 不用 `memory_turn`：直接从书架拿最后一本书（快，但如果书架空了会出错）
- 用 `memory_turn`：先检查书架有没有书，有的话记下位置再拿（慢一点，但安全）

这是**代码健壮性 vs 简洁性**的权衡，参考实现选择了健壮性。
